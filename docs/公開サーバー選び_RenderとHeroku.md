# 公開サーバー選び：Render と Heroku の相性

現在のシステム（Flask + SQLAlchemy + SQLite/PostgreSQL、Google Drive 連携）を公開するときの、**Render** と **Heroku** の相性とコストをまとめます。

---

## 結論（コストを抑えたい場合）

| 目的 | おすすめ |
|------|----------|
| **試運用・コストを抑えたい** | **Render** の無料枠 or 低額プラン |
| **正式運用に切り替えたい** | **Heroku** や VPS などに移行 |

今のコードは **Render とも Heroku とも相性が良い**です。試運用は Render で始め、正式運用になったら Heroku や自前サーバーに移行する構成が現実的です。

---

## 現在のシステムとの相性

### 共通で必要な対応（どちらでも同じ）

| 項目 | 内容 |
|------|------|
| **WSGI** | すでに `wsgi.py` があり、Gunicorn でそのまま利用可能 |
| **DATABASE_URL** | `config.py` で環境変数 `DATABASE_URL` を参照済み。PostgreSQL にすれば本番向け |
| **ファイル保存** | どちらも **エフェメラル**（再デプロイで消える）。領収書は **Google Drive 連携**を使う運用が向く |
| **環境変数** | SECRET_KEY・Google Drive 系を環境変数で渡す想定で済んでいる |

このため、**Render 用に書いた設定は Heroku に持っていってもほぼそのまま**使えます。

## GitHub の位置づけ

**GitHub は「アプリが動くサーバー」ではなく、コードの置き場（リポジトリ）**です。

| 役割 | 説明 |
|------|------|
| **コードの保管** | ソースを GitHub にプッシュしておく。バージョン管理・バックアップになる |
| **デプロイの起点** | Render も Heroku も「GitHub リポジトリと連携」が基本。`git push` すると自動でビルド・デプロイされる |
| **本番で動かす場所** | GitHub 上では動かない。**実際にアプリが動くのは Render / Heroku / VPS など** |

つまり、

- **GitHub** … コードを置く場所（必須の土台）
- **本番の DB** … PostgreSQL（`DATABASE_URL`）
- **領収書などファイル** … Google Drive 連携を前提
- **アプリの実行環境** … Render（試運用）→ Heroku など（正式運用）

という分担になります。GitHub にプッシュしたうえで、Render や Heroku がそのリポジトリを参照してデプロイする、という流れです。詳細なプッシュ手順は `docs/GitHubとVercel公開.md` の「1. GitHub に連携する」を参照してください。

## Render の特徴（コスト重視）

- **無料枠**: Web サービスが **15分無操作でスリープ**。起動に約 1 分かかることがある
- **750 時間/月** まで無料。常時起動は無理だが、個人・試運用なら十分なことが多い
- **無料 PostgreSQL**: 30 日で消えるため、長期なら有料 DB（$7/月）か外部（Neon・Supabase 等の無料 Postgres）を検討
- **有料**: 常時起動 Web サービスは **$7/月** 程度から
- **相性**: Flask + Gunicorn は標準的。Git 連携で自動デプロイ可能

**向いている場面**: まずはお金をかけずに公開して試したい、正式運用前の検証期間。

---

## Heroku の特徴（安定・正式運用向け）

- **無料プラン廃止**（2022年〜）。**Eco Dyno $5/月** + **Postgres Mini $5/月** で月約 **$10〜** が目安
- **相性**: Flask との相性は非常に良い。Buildpack・アドオンが豊富で情報も多い
- **スリープ**: Eco だと 30 分無操作でスリープするが、起動は比較的スムーズ
- **本番運用**: 監視・ログ・スケールのノウハウが多く、**正式運用に切り替えるときの候補**として最有力

**向いている場面**: 試運用を終え、安定して本番として使いたいとき。

---

## 運用方針の例

1. **今（試運用）**  
   - **Render** の無料 or $7/月 でデプロイ  
   - PostgreSQL は Render の有料 DB か、Neon / Supabase の無料 Postgres を `DATABASE_URL` で指定  
   - 領収書は Google Drive 連携を前提にし、サーバーのエフェメラルストレージに依存しない

2. **正式運用に切り替えるとき**  
   - **Heroku** に移行（同じ `DATABASE_URL`・環境変数・Gunicorn 構成でほぼそのまま）  
   - または VPS（さくら・ConoHa 等）で Gunicorn + Nginx 運用

---

## まとめ

- **今のシステムは Render とも Heroku とも相性が良い**（WSGI・環境変数・DB の設計が共通で使える）
- **コストを抑えたいなら Render** で試運用 → **正式運用になったら Heroku や VPS に移行**する流れが現実的です。
- どちらでも、本番では **PostgreSQL**（`DATABASE_URL`）を使い、領収書は **Google Drive** 中心にすると運用が楽です。

Render での具体的なデプロイ手順は `docs/Renderデプロイ.md` を参照してください。
